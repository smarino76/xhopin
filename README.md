# xhopin

**An embedded kernel that runs every application in an isolated VM, with declarative permissions and dynamic loading.**  
Portable (MIPS, soon ARM/RISC‑V), written in C, designed for **reliable and modular** systems.

> ⚠️ **Status**: active development. The foundations (VM, scheduler, loader, abstract console, abstract storage, ramfs) are **working**. Many pieces are still missing (bus drivers, other filesystems, network stack, distributed computing). This README describes the **vision**, what is already implemented, and what is planned.

---

## The problem

In traditional embedded systems (FreeRTOS, Zephyr, Mbed OS):

- A **crash** in any application **brings down the whole system**.
- Applications are **compiled together with the kernel** – to update them you have to reflash the entire firmware.
- Security relies on the **developer’s goodwill** (no runtime permission checks).

## The solution

**xhopin** introduces a completely different model:

- **Hardware‑assisted isolation** (sandbox): every application runs in a **32‑bit VM** with private virtual RAM. If an app crashes, the kernel and other apps **keep running**.
- **Capability‑based security** (Android‑style): each app declares in its manifest the permissions it needs (`CAP_LOGGING`, `CAP_STORAGE`, …). The kernel **blocks any unauthorised access**.
- **Dynamic loading**: applications are independent binary files, loadable at runtime from console, storage, or network. **No firmware reflashing**.
- **Portability**: the kernel is written in standard C; hardware‑specific code is confined to a few hundred lines per architecture.

## Current status (what already works)

- ✅ **VM** with a complete instruction set (arithmetic, logic, shift, memory, branches, function calls, syscall).
- ✅ **Preemptive round‑robin scheduler** (core timer, assembly context switch).
- ✅ **Abstract console** – `console_write`/`read` APIs exist; currently only UART console is implemented (using Harmony drivers).
- ✅ **Abstract storage** – `storage_read`/`write` APIs exist; currently only a **RAM disk** is implemented (no SD, SPI flash, etc.).
- ✅ **Abstract filesystem** – `fs_create`/`read`/`write`/`remove` APIs exist; currently only **ramfs** (in‑memory filesystem) is implemented.
- ✅ **Application loader** – magic, checksum, ABI version checks, task and VM creation.
- ✅ **Interactive shell** (App 0) – commands: `load`, `stop`, `download`, `list`, `help`.
- ✅ **Host assembler** (`vm_asm`) – converts VM assembly into executable binaries.

## What is missing (in progress or planned)

- 🟡 **Bus drivers** (I2C, SPI) – abstractions exist (`arch_bus_i2c.h`, `arch_bus_spi.h`), but generic implementations are missing (currently rely on Harmony for PIC32).
- 🟡 **Other filesystems** (FAT, LittleFS) – only `ramfs` is implemented.
- 🟡 **Persistent storage** (SD card, SPI flash) – only RAM disk.
- 🟡 **Console on other devices** (display, USB) – only UART.
- 🟡 **Configurable security policy** – administrative control over permissions (planned).
- 🟡 **`CALLR` instruction** – indirect function call (useful for function pointers).
- 🔴 **Distributed cluster computing** – under design.
- 🔴 **Network stack** – not started.

## Hardware abstraction – no hardware drivers in the kernel

The kernel **does not contain hardware drivers**. It only provides **abstractions**:

- `console_t` – `write` / `read` operations.
- `storage_dev_t` – `read_block` / `write_block` operations.
- `device_t` – registration and lookup by class.

Real driver implementations are **external** to the kernel. In the current port for **PIC32MX470F512H**, we use drivers generated by **Microchip Harmony** (UART, I2C, SPI, core timer). To port the kernel to a new architecture (ARM, RISC‑V), you only need to implement a small HAL (context switch, interrupt, timer, and the desired peripheral drivers).

## Future vision: distributed cluster computing

We are designing a feature that will allow a VM (master) to **offload heavy computations to other CPUs in the cluster** in an **asynchronous and transparent** way.

The API will be simple:

- `ACTION_FORK` – pass a function (bytecode) and its arguments; returns a **job ID**.
- `ACTION_JOIN` – retrieve the result (blocking or non‑blocking).
- The kernel will decide whether to run the job **locally** (by creating a task) or **remotely** (by sending it to a network worker).

**Benefits**:

- The application does not know whether the computation is local or remote.
- Adding workers scales performance linearly.
- If a worker fails, the job can be reassigned.

> *This will make xhopin one of the few embedded kernels with native **cluster computing** support.*
